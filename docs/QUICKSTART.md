# Quickstart

## Install

```bash
bash <(curl -fsSL https://github.com/jiwen77/relaypilot/raw/refs/heads/main/install-relaypilot.sh)
```

Interactive terminals enter the menu automatically after install. Reopen it with
`relaypilot` or `relaypilot menu`:

```text
RelayPilot
1) Hub 模式
2) Agent 模式
3) 本机服务
0) 退出
```

Use **Hub 模式** on the Hub machine. Use **Agent 模式** on transit/landing
machines; configure this machine as transit or landing, then paste the matching
invite generated by the Hub. Automation can still stay one line:

```bash
bash <(curl -fsSL https://github.com/jiwen77/relaypilot/raw/refs/heads/main/install-relaypilot.sh) \
  --enroll 'PASTE_INVITE'
```

The installer uses a Go binary when a matching release asset exists. Bash keeps
the one-command UX. Release builds publish `relaypilot_*` assets only.
RelayPilot is Go-only at runtime; common Telegram helpers are handled by the Go
core when the binary is available.

## State migration

RelayPilot uses `/etc/relaypilot` by default. To move state between directories:

```bash
relaypilot migrate-state --from /path/to/old-state --to /etc/relaypilot --dry-run
relaypilot migrate-state --from /path/to/old-state --to /etc/relaypilot
```

The migration only copies files and keeps the source directory as a backup.

## Update

Use the local menu on any installed machine:

```text
relaypilot
3) 本机服务
6) 更新 RelayPilot
```

Or run automation:

```bash
relaypilot update
relaypilot update --version v0.1.2 --restart-services
```

This updates the entrypoint and Go core. Restart Hub/Agent/Bot services to make
already-running daemons use the new version.

## Agent data-plane setup

Pasting a Hub invite enrolls the Agent into the control plane. The interactive join wizard then detects the invite role and offers the matching local sing-box setup.

Transit machine:

```text
relaypilot
2) Agent 模式
3) 粘贴 Hub invite 并安装 Agent 服务
# role=transit -> 初始化/更新 Reality 入口
```

Shortcut command for standalone/automation:

```bash
bash relaypilot.sh transit-init-reality
```

It creates or updates the VLESS Reality inbound and generates Reality key/short_id when omitted.

Landing machine:

```text
relaypilot
2) Agent 模式
3) 粘贴 Hub invite 并安装 Agent 服务
# role=landing -> 安装/更新 Shadowsocks
# 前置机房/端口转发 -> 6) 公网入口
```

Shortcut command:

```bash
bash relaypilot.sh landing-install-ss
```

It writes the landing Shadowsocks inbound and stores the endpoint secret under `/etc/relaypilot/endpoints/<name>.json`. With Hub linking you do not copy this endpoint by hand; the Hub fetches it from the landing agent and sends it to the transit agent as a protected task payload.

If the landing service is reached through a fronting IP/domain with port
forwarding, store the externally reachable address on the Agent:

```bash
relaypilot public-entry-set --use shadowsocks --name hk --host front.example --public-port 443 --local-port 2443
# Mesh/WireGuard only if UDP is forwarded too:
relaypilot public-entry-set --use wireguard --name hk --host front.example --public-port 51820 --local-port 50123 --network udp
```

Manual standalone binding remains available:

```text
relaypilot
2) Agent 模式
1) 配置中转节点
2) 绑定落地 endpoint
```

## Hub server

Use a Hub when you have more than one transit/landing agent.

```bash
bash relaypilot.sh hub
```

On the Hub, initialize native TLS and create a short-lived agent invite:

```bash
# Recommended menu path:
relaypilot
# 1) Hub 模式
#   1) 初始化 Hub 服务
#   2) 生成 Agent invite

# Concise command path:
relaypilot hub-quick-setup
relaypilot hub-enroll

# Non-interactive equivalent:
bash relaypilot.sh hub-create-enroll-code \
  --agent-id transit-hk \
  --role transit \
  --ttl 10m

# Use --text when a human will copy the install command.
bash relaypilot.sh hub-create-enroll-code \
  --agent-id transit-hk \
  --role transit \
  --ttl 10m \
  --text

# Optional: use a domain/IP explicitly.
bash relaypilot.sh hub-create-enroll-code \
  --public-host hub.example \
  --agent-id transit-hk \
  --role transit \
  --ttl 10m
TG_DRY_RUN=1 bash relaypilot.sh bot register --hub
```

Invite creation also creates/updates the Agent as `待接入` on the Hub. If a
target install fails or the invite expires, rerun the same `--agent-id` to get a
fresh short-lived single-use invite; pending role/name/labels are reused when
omitted.

For public-Internet control-plane traffic, run the Hub with RelayPilot native
HTTPS/mTLS instead of exposing plain HTTP:

```bash
bash relaypilot.sh hub-daemon \
  --host 0.0.0.0 \
  --port 8443 \
  --tls-cert /etc/relaypilot/hub-tls/hub.crt \
  --tls-key /etc/relaypilot/hub-tls/hub.key \
  --client-ca /etc/relaypilot/hub-tls/ca.crt \
  --require-client-cert
```

On the agent host, paste the `install_command` printed by `hub-create-enroll-code`. If RelayPilot is already installed, use Agent mode or the invite manually; both write the agent token, pinned Hub CA, client certificate/key, and `agent-enrollment.json` locally. The interactive Agent mode also offers the local data-plane setup for the invite role before installing the poll service:

```text
relaypilot
2) Agent 模式
1) 配置中转节点
2) 配置落地节点
3) 粘贴 Hub invite 并安装 Agent 服务
```

```bash
relaypilot agent enroll --invite 'PASTE_INVITE' --install-service
relaypilot transit-init-reality   # transit data plane
relaypilot landing-install-ss     # landing data plane
relaypilot agent ip-mode --mode dynamic --public-ip-interval 600  # optional
relaypilot public-entry-set --use shadowsocks --name hk --host front.example --public-port 443 --local-port 2443  # optional forwarding
relaypilot agent poll-once \
  --enrollment-file /etc/relaypilot/agent-enrollment.json
```

Keep the default `static` IP mode for stable IP/domain nodes. `dynamic` adds a
short public-IP probe every 10 minutes by default and only records the result on
Hub.

When both the transit and landing agents are online, link them from the Hub:

```text
relaypilot
1) Hub 模式
5) 串联中转/落地
```

Equivalent command:

```bash
relaypilot hub-link transit-hk landing-hk hk
# Default is direct landing access. Use mesh for a dedicated WireGuard /30:
relaypilot hub-link transit-hk landing-hk hk --mode mesh
```

This queues a control-plane flow: the landing agent exports its full endpoint to
Hub, then Hub queues a bind task to the transit agent. You do not copy endpoint
JSON by hand. In `direct` mode the transit dials the landing endpoint server
directly. In `mesh` mode Hub provisions a dedicated WireGuard transit↔landing
link first and then points the transit outbound at the landing overlay IP; it
requires `wg-quick` on both agents and does not use Hub as a data relay.
When a landing Agent has public-entry records, Hub uses those forwarded
host/ports for exported Shadowsocks endpoints and WireGuard mesh peer endpoints.

In Hub mode `/relaypilot_status` is Hub-only. Use `/relaypilot_status all` or
`/relaypilot_status transit-hk` when you explicitly want to target agents.
Short aliases such as `/status all` still work.

Use `/relaypilot_topology` to see the human-facing Transit -> Landing tree.

## Low-resource services

Before installing background services, inspect the selected limits:

```bash
bash relaypilot.sh resource-profile
```

For very small NAT boxes, force the tiny profile:

```bash
RELAYPILOT_PROFILE=tiny bash relaypilot.sh agent install-service \
  --enrollment-file /etc/relaypilot/agent-enrollment.json
```

Profiles are install-time defaults, not a runtime self-tuner. Exact environment
variables such as `AGENT_SERVICE_MEMORY_MAX=80M` or
`AGENT_SERVICE_CPU_QUOTA=20%` override the profile.
